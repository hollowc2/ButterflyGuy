Entry pipeline

  1. Trading day, time window, chain freshness.
  2. Risk state and live buying power.
  3. Spot, prior close, session open, VIX, current 0-DTE chain.
  4. Gap/regime override or direction/bias signal.
  5. Shared select_entry_candidate logic.
  6. Candidate + parity audit persistence.
  7. Paper simulation or durable live intent + Schwab order.
  8. Trade persistence, risk/metrics update, Discord entry chart.

Position and exit pipeline

  1. Compute fly bid/mark, PnL, time regime, quote quality.
  2. Accept/reject peak updates and persist monitoring legs.
  3. Evaluate peak trailer or profit protector.
  4. Store signal, mark parity, and pending-exit evidence.
  5. Execute exit; retry while the trade remains OPEN if unfilled.
  6. Close on fill or supported cash-settlement evidence.

RiskEngine guards

Daily and weekly loss, daily trade count, position size, consecutive loss warning, minimum buying power, and fail-safe behavior when balances cannot be read.

OrderManager guarantees

Three-leg order payloads, paper/live separation, broker status parsing, partial/terminal/ambiguous fill errors, intent linkage, and entry/exit ladder behavior.

Recovery guarantees

Open trade and peak survive restart. Metrics and risk state are rebuilt. Live broker positions, working orders, intents, and transaction settlements must agree before entry proceeds.

Audit chain

market snapshot → candidate rows → selection parity event → broker intent/fill evidence → butterfly trade → monitoring legs → exit parity event → closed trade/PnL

Database-enforced invariants

One OPEN trade per underlying/date, one active ENTRY intent per underlying/date, and one active EXIT intent per trade. Leg quotes cascade with their trade; filled broker intent can be linked back to the trade.

Network

All asset apps attach to the external monitoring_net, where TimescaleDB, Prometheus, Grafana, and Alertmanager are expected to be reachable.

Container hardening

Read-only filesystem, tmpfs at /tmp, no-new-privileges, all capabilities dropped, and explicit mounts for tokens, config, and chain data.

Prometheus + Grafana

Prometheus scrapes asset and candidate metrics. The candidate dashboard reads its separate TimescaleDB and tracks the 20-trade forward gate.

Configuration boundary

Asset behavior lives in YAML; secrets and runtime credentials remain in .env and tokens.json. Never copy their values into reports or this map.

Notifications

SPX can send Discord entry/exit charts. Telegram carries risk/operational warnings. Alertmanager receives critical live reconciliation, ambiguity, and settlement conditions.

Runtime command

docker compose -f infra/docker-compose.yml up -d starts primary SPX. The paper twin requires the explicit spx-candidate profile; NDX and XSP retain their own profiles.

Non-negotiable safety invariants

  • Do not change live/paper mode, account checks, routing, token handling, loss limits, or trade caps unless directly asked.
  • State the risk before any command that could place an order, call a Schwab write API, or alter live services.
  • Preserve mark-price paper fills and fail-closed broker reconciliation.
  • Never print or summarize secret values from .env or tokens.json.
  • Confirm graph-derived relationships in code/tests: the report has many isolated nodes and inferred edges.
